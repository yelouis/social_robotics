# AI Task Breakdown: Dehydrated Export and Distribution

## Objective
Package the sum total of all the extracted social metadata generated by the Social Feature Layers into a clean format to be posted securely on open data platforms such as Hugging Face.

---

## The "Dehydration" Rule
To strictly adhere to AI ethics, safety frameworks, and copyright logic enforced on varying video datasets (like Ego4D, Epic-Kitchens, etc.), **we do not redistribute source pixels**.

All resulting export files **must be completely dehydrated**. This means:
- No `.mp4`, `.avi`, or `.png` frame exports.
- No inclusion of isolated audio tracks.
- Focus strictly on the extracted parameters: kinematics, layer decisions, VLM contexts, timestamps, and confidence scores.

---

## Layer-Result Aggregation
Before export, the individual layer output JSONs must be merged into a single master table. Each Social Feature Layer (03a, 03b, etc.) produces its own result file keyed by `video_id`. The aggregation step is responsible for joining these into a unified record per video.

### Aggregation Logic
1. **Scan** the results directory for all `03*_result.json` files.
2. **Outer-join** on `video_id`: Every video that appears in *any* layer's output gets a row. If a layer did not process a given video (or failed), its columns are filled with `null`/`NaN`.
3. **Flatten nested structures**: Per-person arrays (from 03a, 03b) should be flattened into the aggregate summary fields for the parquet columns. The full per-person detail can be preserved in a JSON-encoded string column for research use.
4. **Attach manifest metadata**: Join in the `task_label`, `source_dataset`, `duration_sec`, and `fps` from `filtered_manifest.json` so the exported dataset is self-describing.

### Expected Intermediate Output
A `merged_social_features.json` (or SQLite DB) containing one record per `video_id` with all layer features as columns, ready for conversion to Parquet.

---

## Required Package Structure for Export

### 1. Rehydration Instructions (`rehydrate_dataset.py`)
To make the dataset useful to the research community, you must clearly write instructions or include a python script detailing how a downstream user can map these social features *back onto* their own valid, legally obtained copies of the source videos (Ego4D). 
- Reference exact external dataset ID numbers.
- Provide mapping schemas (e.g. "We extracted `flinch: true` at timestamp `14.5s` for Ego4D UUID `XYZ-123`").

### 2. The Result File (`social_metadata.parquet`)
Merge the final state of the layered results database into a highly compressed Apache Parquet file format. Parquets are optimized for loading into Pandas or Hugging Face `/datasets` directly. Ensure NaN/Missing columns (due to a video not triggering a specific layer, or a layer failing) are handled cleanly.

### 3. Posting to Hugging Face
If the local `result_file` review looks good, write an automated script leveraging the `huggingface_hub` Python package to initialize the dataset repository, upload the `social_metadata.parquet`, and populate a generated `README.md` Dataset Card outlining the total number of layers utilized to create the signals.
