# AI Task Breakdown: Dehydrated Export and Distribution

## Objective
Package the sum total of all the extracted social metadata generated by the Social Feature Layers into a clean format to be posted securely on open data platforms such as Hugging Face.

---

## The "Dehydration" Rule
To strictly adhere to AI ethics, safety frameworks, and copyright logic enforced on varying video datasets (like Ego4D, Epic-Kitchens, etc.), **we do not redistribute source pixels**.

All resulting export files **must be completely dehydrated**. This means:
- No `.mp4`, `.avi`, or `.png` frame exports.
- No inclusion of isolated audio tracks.
- Focus strictly on the extracted parameters: kinematics, layer decisions, VLM contexts, timestamps, and confidence scores.

---

## Layer-Result Aggregation
Before export, the individual layer output JSONs must be merged into a single master table. Each Social Feature Layer (03a, 03b, etc.) produces its own result file keyed by `video_id`. The aggregation step is responsible for joining these into a unified record per video.

### Aggregation Logic
1. **Scan** the results directory for all `03*_result.json` files.
2. **Outer-join** on `video_id`: Every video that appears in *any* layer's output gets a row. If a layer did not process a given video (or failed), its columns are filled with `null`/`NaN`. Missing `video_id` records are skipped safely.
3. **Flatten nested structures & Extract Summaries**: 
   - **Schema A** (`aggregate` / `per_person`) and **Schema B** (`tasks_analyzed`) are flattened.
   - A centralized `LAYER_SUMMARY_REGISTRY` maps each layer to specific scalar fields (e.g., `avg_task_score`, `max_proxemic_confidence`). These are extracted, aggregated (mean, max, max_abs, any), and surfaced as dedicated Parquet columns.
   - The full nested arrays are preserved in JSON-encoded string columns (`*_raw`) for research use.
4. **Top-Level Provenance Flattening**: Top-level dictionaries (such as `processing_meta`) are flattened by one level so that critical provenance details are exposed as Parquet columns (e.g., `<layer>_<dict_key>_<sub_key>`).
5. **Attach manifest metadata**: Join in the `task_labels` (parsed from the `identified_tasks` array), `source_dataset`, `duration_sec`, and `fps` from `filtered_manifest.json` so the exported dataset is self-describing.
6. **Embed export metadata**: 
   - We use a dual-export approach. The dataset is exported as `social_metadata.parquet`, and provenance data is serialized into a companion `export_metadata.json` to avoid issues with Parquet metadata serialization across different readers.
   - Every exported dataset includes a top-level metadata block for traceability:
     ```json
     {
       "schema_version": "1.0.0+a3f9c2",
       "export_timestamp": "2026-05-18T00:00:00Z",
       "active_layers": ["03a", "03b", "03c", "03d", "03e", "03f", "03g"],
       "pipeline_git_sha": "abc1234"
     }
     ```
   - `schema_version` follows an additive-only protocol. The minor version can be bumped, and a SHA-256 hash suffix of the sorted column names (`+<short_hash>`) is appended. This allows downstream consumers to mechanically detect schema drift between exports.

### Memory Management & Dask Fallback
To prevent Out-Of-Memory (OOM) failures when merging massive Parquets on a single node:
- **Dynamic Memory Check**: The aggregator computes an `effective_size` of the source JSONs (multiplied by a `PANDAS_INFLATION_FACTOR = 8` to account for object overhead).
- **Fallback Trigger**: If the effective size exceeds the available memory headroom (70% on hosts with ≥ 48 GB RAM, otherwise 50%), the pipeline switches from Pandas to a Dask-based chunked outer-join.
- **Partitioning & Streaming**: Dask partitions are sized dynamically. When streaming is requested (by passing `output_parquet_path`), the chunked DataFrame is written directly to a partitioned Parquet directory, bypassing full in-memory materialization.
- **Recalibration**: The `_compute_inflation_factor` helper can measure the empirical inflation ratio for a completed export, allowing operators to adjust the factor if the schema shifts toward more memory-intensive data types.

### Expected Intermediate Output
A `merged_social_features.json` (or SQLite DB) containing one record per `video_id` with all layer features as columns, ready for conversion to Parquet.

## Required Package Structure for Export

### 1. Rehydration Instructions (`rehydrate_dataset.py`)
To make the dataset useful to the research community, you must clearly write instructions or include a python script detailing how a downstream user can map these social features *back onto* their own valid, legally obtained copies of the source videos (Ego4D). 
- Reference exact external dataset ID numbers.
- Provide mapping schemas (e.g. "We extracted `flinch: true` at timestamp `14.5s` for Ego4D UUID `XYZ-123`").

### 2. The Result File (`social_metadata.parquet`)
Merge the final state of the layered results database into a highly compressed Apache Parquet file format. Parquets are optimized for loading into Pandas or Hugging Face `/datasets` directly. Ensure NaN/Missing columns (due to a video not triggering a specific layer, or a layer failing) are handled cleanly.

### 3. Posting to Hugging Face
If the local `result_file` review looks good, write an automated script leveraging the `huggingface_hub` Python package to initialize the dataset repository, upload the `social_metadata.parquet`, and populate a generated `README.md` Dataset Card outlining the total number of layers utilized to create the signals.

## Verification & Validation Check
To guarantee the export structure adheres to safety standards and schema requirements:
- **Singular Video Test**: Load the final `social_metadata.parquet` into a Pandas DataFrame inside a standalone Jupyter notebook or script. Select one random row (a single video) and attempt to cleanly follow the `rehydrate_dataset.py` instructions to rebuild the social context. If any keys are mismatched, it fails.
- **Batch Test**: Run an automated validation script across the entire merged DataFrame. Assert that there are absolutely zero raw visual bytes (no images/video buffers stored). This check explicitly scans for Python `bytes`, `numpy.ndarray` buffers, base64-encoded image strings, and raw local file path references. Validate that the row count matches exactly the number of clips successfully parsed, and perform a null-check analysis. Verify that this Pandas aggregation operates within the data limits of our **Mac Studio (M4 Max, 64 GB unified memory)** when reading multi-gigabyte Parquets into memory.
